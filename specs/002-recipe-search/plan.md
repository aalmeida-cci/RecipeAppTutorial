# Implementation Plan: Recipe Search

**Branch**: `002-recipe-search` | **Date**: 2026-05-27 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-recipe-search/spec.md`

## Summary

Add a fully reactive recipe search screen that queries the existing local SQLDelight `Recipe` table using `LIKE` across `title`, `description`, and `ingredients`. Results update in real time as the user types (300 ms debounce, `flatMapLatest` cancellation). The screen is already reachable from the Feed search bar and is wired into the existing navigation graph; this plan fills in all layers from the SQLDelight query up through the Compose UI.

## Phase 0: Research

*See [research.md](research.md) for full decisions. Summary:*

- **Reactive search**: `MutableStateFlow<String>` → `.debounce(300)` → `.flatMapLatest { searchRepo.searchRecipes(it) }` — standard KMP coroutines pattern, no new dependency.
- **LIKE query**: SQLDelight parameterised query using `lower()` for case-insensitive matching. SQLite `lower()` is ASCII-only which is sufficient for recipe titles; a full Unicode locale-aware collation is out of scope.
- **Result reuse**: `RecipeItem` domain entity covers all fields needed for result rows (imageUrl, title, duration, rating, id). No new domain entity.
- **No RemoteDataSource**: Search is local-only per spec; no `RemoteDataSource` added to `search/data/datasources/`.
- **State model**: Sealed interface with 5 states — `Idle`, `Loading`, `Success(results)`, `Empty`, `Error(throwable)` — maps cleanly to the 4 spec-required UI states (Idle collapses into the "initial empty list" state defined in clarifications).

## Phase 1: Design

### Layer-by-Layer Specification

#### 1. SQLDelight Query — `RecipeEntity.sq`

Add after existing queries:

```sql
searchRecipes:
SELECT * FROM Recipe
WHERE lower(title)       LIKE ('%' || lower(:query) || '%')
   OR lower(description) LIKE ('%' || lower(:query) || '%')
   OR lower(ingredients) LIKE ('%' || lower(:query) || '%');
```

- `:query` is the raw user text (already blank-guarded in the ViewModel before this is called).
- `lower()` provides case-insensitive ASCII matching — sufficient for recipe names.
- `ingredients` is stored as a comma-separated string by `ListOfStringsAdapter`; `LIKE` substring match against the serialised form is correct for this use case.

**Impl** (`search/data/repositories/SearchRepositoryImpl.kt`):

```kotlin
    private val localDataSource: SearchLocalDataSource
) : SearchRepository {
    override suspend fun searchRecipes(query: String): Result<List<RecipeItem>> = try {
        Result.success(localDataSource.searchRecipes(query))
    } catch (e: Exception) {
        Result.failure(e)
    }
}
```

#### 5. SearchUiState

```kotlin
sealed interface SearchUiState {
    data object Idle    : SearchUiState   // query is blank — show nothing
    data object Loading : SearchUiState   // query in flight
    data class  Success(val recipes: List<RecipeItem>) : SearchUiState
    data object Empty   : SearchUiState   // query returned 0 results
    data class  Error(val throwable: Throwable) : SearchUiState
}
```

#### 6. SearchViewModel

```kotlin
class SearchViewModel(private val searchRepository: SearchRepository) : ViewModel() {

    private val _query = MutableStateFlow("")
    val query: StateFlow<String> = _query.asStateFlow()

    val searchUiState: StateFlow<SearchUiState> = _query
        .debounce(300L)
        .flatMapLatest { q ->
            if (q.isBlank()) {
                flowOf(SearchUiState.Idle)
            } else {
                flow {
                    emit(SearchUiState.Loading)
                    searchRepository.searchRecipes(q).fold(
                        onSuccess = { results ->
                            emit(if (results.isEmpty()) SearchUiState.Empty else SearchUiState.Success(results))
                        },
                        onFailure = { emit(SearchUiState.Error(it)) }
                    )
                }
            }
        }
        .stateIn(
            scope = viewModelScope,
            started = SharingStarted.WhileSubscribed(5_000),
            initialValue = SearchUiState.Idle
        )

    fun onQueryChange(newQuery: String) { _query.value = newQuery }
}
```

- `flatMapLatest` automatically cancels the previous `flow { }` block when `_query` emits.
- `debounce(300)` prevents a DB query on every keystroke.
- `SharingStarted.WhileSubscribed(5_000)` matches the Jetbrains/Google ViewModel + StateFlow best practice for this stack.

#### 7. SearchScreen.kt

Structure (all in `commonMain`):

```
SearchRoute(viewModel, navigateBack, navigateToDetail)
  └── SearchScreen(query, onQueryChange, uiState, navigateBack, navigateToDetail)
        ├── Scaffold
        │     topBar: SearchTopBar(title="Search Recipes", onBack=navigateBack)
        │     content:
        │       Column
        │         SearchInputField(query, onQueryChange)   ← rounded, search icon, placeholder
        │         when (uiState)
        │           Idle    → (empty — nothing rendered)
        │           Loading → Loader()                     ← reuse existing
        │           Success → RecipeSearchResultList(recipes, navigateToDetail)
        │           Empty   → EmptyResultsContent()
        │           Error   → ErrorContent()               ← reuse existing
```

Key Compose rules:
- `Modifier` first optional param in every composable.
- `collectAsStateWithLifecycle()` (not `collectAsState()`).
- `query` and `onQueryChange` are hoisted out of the composable — ViewModel owns state, Screen is stateless.
- Result rows are a `LazyColumn` with `items(recipes, key = { it.id })`.
- Each `SearchRecipeRow` composable: `AsyncImage` (80dp × 80dp, rounded) + Column(title, Row(duration icon + rating icon)).
